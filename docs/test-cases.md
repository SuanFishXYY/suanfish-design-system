# 算鱼设计系统 v3.2.0 · 测试用例集

> 5 个真实任务 · 完整走 10 站流程 · 给出**预期召唤的圣人清单 + 三段式输出样例**, 供后续手动或自动化验证可重复对照。

---

## 🧭 使用说明

每个测试用例包含:
- **任务输入** — 用户原话
- **预期 Tier 0 八圣人议会输出** — 八圣人各自的初步审查
- **预期 bench-matcher 召唤清单** — 从 301 板凳中应被召唤的 N 位 (含评分维度)
- **预期 N 位三段式样例** — 理论依据 / 设计方向 / 改造动作
- **预期 Path 路径** — A-G 哪条
- **预期主导 agent** — 下游接手者

如 skill 实际运行与本文档预期偏差较大 → 视为 regression, 需调查。

| 🪙 dialectician | 主矛盾: D1 简洁⟷可发现 + D5 数据⟷直觉 · 倾向 D5 数据侧 | — | major |
| 📜 historian | 当前 E5 玻璃拟态? 建议 E4 扁平化 / E7 AI-native | R14? | minor |
| 🔭 futurist | L4 反馈循环律: dashboard 必须 ≤200ms 响应 | — | info |
| 🪷 wuwei-master | KPI 卡冗余风险 → 默认显示 ≤ 5 个 | R19 警告 | major |

| 🔬 debunk-auditor | 警惕"AI 智能评分"伪指标 | R23 警告 | critical |

### 三段式样例 (节选 · 王充)

> **📚 理论依据**: 王充《论衡》"疾虚妄"。dashboard 最大病灶是用复杂图表掩盖数据稀疏。
>
> **🎯 设计方向**: 反炫技 · 反图表通胀 · 每个图表必须可问"这是真数据还是装的?"
>
> **🔧 改造动作**:
> 1. 删除所有动效装饰 (扫描线 / 流光)
> 2. 数据点 < 5 的图表换成表格
> 3. 每个指标加"数据源 + 更新时间"小字注释

### 下游路径

## TC-03 · AI 聊天对话 UI (含 streaming / tool-call)

### 任务输入
> "做一个 AI 助手聊天界面, 要支持 streaming 输出 + tool-call 可视化 + 引用来源"

### Tier 0 议会要点

| 圣人 | 观察 | 严重度 |
| --- | --- | --- |
| 🪙 dialectician | D2 自动化⟷掌控 · 倾向掌控 (可中断 streaming) | major |
| 🔭 futurist | L5 模态融合律: 必须支持图文混合输出 | major |
| 🔬 debunk-auditor | streaming 必须真实, 不许 fake-stream 已就绪文本 | R23 强警告 | critical |

### bench-matcher 预期召唤 (7 位)

| 圣人 | score | 备注 |
| --- | --- | --- |
| #225 王充 (疾虚妄 · 真 streaming) | 9.5 | critical 任务 |
| #086 维纳 (控制论 · 反馈) | 8.8 | 已用 ✅ |
| #084 玛丽·道格拉斯 (秩序) | 8.0 | 工具调用分类 |
| #161 哈贝马斯 (沟通理性) | 7.9 | 引用透明 |
| #058 福柯 (知识考古) | 7.7 | 引用必须可追 |
| #056 杜威 (互动学习) | 7.5 | 用户可介入 |
| #210 庄子 (允许打断) | 7.2 | 控制权 |

### 三段式样例 (节选 · 福柯)

> **📚 理论依据**: 福柯《知识考古学》, 任何"权威输出"必须能溯源到话语形成的历史条件。
>
> **🎯 设计方向**: 每个引用必须可点击 → 跳转到真实来源, 不允许"凭空引用"。
>
> **🔧 改造动作**:
> 1. citation 卡片必须含 URL + 截取时间 + 章节锚点
> 2. 不可点击的引用 → 标 "no link" 而非装作能点
> 3. 多源引用按可信度排序 (官方 > 媒体 > 论坛)

### 下游路径
**Path C 聊天 + Path G AI-native overlay** → `conversation-director` 主导

## TC-04 · 移动端表单 (注册流程)

### 任务输入
> "做一个移动端注册流程, 6 个字段, 要支持表单一步步引导填写"

### Tier 0 议会要点

| 圣人 | 观察 | 严重度 |
| --- | --- | --- |
| 🪷 wuwei-master | 6 字段能否砍? 哪些可以异步收集 | R19 警告 | major |
| 🐢 perspectivist | 新手 vs 老手 vs 第三方注册, 多 persona | info |
| 🌐 holism-strategist | 每步状态保留 (不丢数据) | major |

### bench-matcher 预期召唤 (4 位)

| 圣人 | score |
| --- | --- |
| #092 老子 (字段砍) | 9.0 |
| #157 西蒙·维伊 (注意力的伦理 · 不浪费用户时间) | 8.5 |
| #018 亚里士多德 (中庸 · 不要太长不要太短) | 7.8 |
| #245 颜元 (习行实践 · 一步一行) | 7.5 |

### 三段式样例 (节选 · 西蒙·维伊)

> **📚 理论依据**: 西蒙·维伊《等待上帝》, 注意力是一种伦理资源, 浪费用户注意力等同剥夺。
>
> **🎯 设计方向**: 6 字段必须减到 ≤ 3 个必填, 其余异步补全。
>
> **🔧 改造动作**:
> 1. 强制必填: 手机号 + 验证码 + 密码 (3 个)
> 2. 异步补全: 头像 / 昵称 / 邀请码 (3 个 · 注册后引导)
> 3. 每步显示进度 "1/3" 而不是 "1/6"

### 下游路径
**Path E 移动** → `ui-architect + responsive-strategist` + `wizard-designer`

## TC-05 · 危险操作二次确认弹窗

### 任务输入
> "做一个删除账号的二次确认弹窗 · 不能误触发"

### Tier 0 议会要点

| 圣人 | 观察 | 严重度 |
| --- | --- | --- |
| 🪙 dialectician | D6 即时⟷深思 · 强烈倾向深思 | critical |
| 🪷 wuwei-master | 反向 R19: 这次需要"加摩擦" 不是删 | info |
| 🔬 debunk-auditor | 不要伪装成温柔提示 (这是真的危险) | critical |

### bench-matcher 预期召唤 (4 位)

| 圣人 | score |
| --- | --- |
| #003 康德 (绝对命令 · 用户应被尊重为目的) | 9.5 |
| #115 阿伦特 (责任的不可推卸) | 8.8 |
| #044 韩非 (法家 · 重赏严罚 / 清晰后果) | 8.2 |
| #225 王充 (疾虚妄 · 不许假装严重) | 7.8 |

### 三段式样例 (节选 · 阿伦特)

> **📚 理论依据**: 阿伦特《人的境况》, 行动一旦发生, 后果不可推卸。删除是不可逆行动。
>
> **🎯 设计方向**: 强调"这个动作不可恢复" + 让用户主动承担 (输入用户名而非点按钮)。
>
> **🔧 改造动作**:
> 1. 弹窗标题: "永久删除账号 — 此操作不可恢复"
> 2. 让用户输入完整用户名才能激活删除按钮
> 3. 删除按钮默认置灰, 输入正确才变红
> 4. 提供"导出数据"前置入口
> 5. 删除后无 toast 庆祝, 直接登出到首页

### 下游路径
**Path B 稳态** → `modal-craftsman` 主导

## 📊 测试用例横向对比

| TC | 任务类型 | Path | 召唤圣人数 | 召唤中/西比 | 主导 agent |
| --- | --- | --- | --- | --- | --- |
| TC-01 | dashboard | B | 6 | 4 中 / 2 西 | data-viz-engineer |
| TC-02 | onboarding | A | 5 | 2 中 / 3 西 | onboarding-director |
| TC-03 | chat | C+G | 7 | 2 中 / 5 西 | conversation-director |

| TC-05 | modal-confirm | B | 4 | 2 中 / 2 西 | modal-craftsman |

**观察**:
- 不同任务召唤的圣人**没有重复完全相同的组合** (验证动态匹配)
- 中西比例不固定 · 由任务性质决定 (数据/中国偏多 · 哲学伦理/西方偏多)
- 召唤数 4-7 位 · 符合 bench-matcher 设计的 3-9 区间

## 🧪 如何使用本测试集

### 手动验证
1. 在 Claude Code 装好 plugin (`/plugin install suanfish-design-system`)
2. 复制 TC-XX 的 "任务输入" 原话发给 skill
3. 对比实际输出与本文档"预期"
4. 偏差 > 30% → 视为 regression, 提 issue

### 自动化验证 (未来 v3.3 计划)
- 写 `scripts/verify-test-cases.js`
- 调用 skill API 跑 TC-01 至 TC-05
- 提取实际召唤圣人清单 vs 预期, 计算召回率
- 召回率 < 60% 报警
