# ✍️ 案例 · copy-writer

哲学锚点：**维特根斯坦 ·「语言的边界即世界的边界」**

---

## ✅ Case-C1 · Mailchimp · 「High fives!」(发送成功)

**场景**：邮件群发成功

**决策**：用 "High fives! Your campaign is sent." 代替 "Success: Email sent."

**哲学依据**：
- `[Canon-D4]`：让用户理解发生了什么 → "sent" 比 "Success" 更直接
- 「High fives!」让庆祝瞬间真实 → 但克制（不滥用）

**注意**：仅在 Mailchimp 的品牌人格下成立。换到企业级 / 严肃产品 → 不适用。

---

## ✅ Case-C2 · Stripe · 「Pay $29.00 to Acme Inc.」

**场景**：支付按钮

**决策**：按钮文案直接显示金额 + 收款方，而不是"确认支付"。

**哲学依据**：
- `[Canon-D6]`：诚实——用户点击前知道确切发生什么
- `[维特根斯坦]`：用户语言世界里"$29.00 to Acme"比"确认"更精确
- 防止误点 + 增强信任

---

## ✅ Case-C3 · Slack · 「You haven't sent any messages yet」

**场景**：新用户加入工作区，自己消息列表为空

**决策**：用"你还没有发送任何消息"代替"暂无数据"

**哲学依据**：
- 指明主体（你）
- 指明动作（发送消息）
- 让用户立刻知道"我可以做什么"

---

## ❌ Case-C4 · 「亲，您好哦~小主~」

**场景**：客服 / 系统提示

**违反**：
- `[Canon-D6]`：装亲密 = 不诚实
- `[Canon-D5]`：刺眼 = 不隐而不见
- B 端 / 严肃产品语境完全错位

**修正方向**：
- 中性 + 直接："你好。" 或直接进入主题，不寒暄

---

## ❌ Case-C5 · 「确认提交」按钮 + 「确认」二级弹窗

**场景**：表单提交

**违反**：
- "确认提交" → 用户不知道提交后会发生什么
- 又来一个"确认" → 完全冗余
- 维特根斯坦：用户的语言世界里"确认"是空概念

**修正方向**：
- 按钮直接表述结果："创建工单" / "发送审批" / "保存草稿"
- 不需要二级确认（除非不可逆，则改为"输入名称确认")

---

## ❌ Case-C6 · 错误提示「服务器开小差了」

**场景**：API 500 错误

**违反**：
- "开小差" = 拟人化 + 推卸责任
- 不告诉用户发生了什么 / 怎么办
- `[Canon-D6]`：极度不诚实

**修正方向**：
- "暂时连不上服务（503）。请稍后重试，或 [刷新]。"
- 或更明确："请求失败。错误码 503。我们的工程师已收到通知。"

---

## 文案优化清单

| 原句 | 优化后 |
| --- | --- |
| 确认提交 | 创建工单 / 发送审批 / 保存草稿（说出动作结果） |
| 暂无数据 | 还没有 X。[创建第一个？] |
| 操作成功 | X 已创建 / X 已发送 / X 已保存（说出对象） |
| 请稍后再试 | 暂时连不上。我们会自动重试，或 [手动刷新]。 |
| 您确定要删除吗？ | 删除 X？（标题）+ [撤销] 后置 |
| 加载中... | 正在加载 X...（说出对象） |
| 服务器开小差 | 请求失败（503）。[重试] |

> **铁律**：每句文案问 3 个问题——
> 1. 主语是谁？（你 / 系统 / 我们）
> 2. 动作是什么？（具体动词）
> 3. 用户下一步能做什么？（CTA）
