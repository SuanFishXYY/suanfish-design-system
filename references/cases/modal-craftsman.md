# 🪟 案例 · modal-craftsman

哲学锚点：**奥卡姆剃刀 ·「如无必要，勿增弹窗」**

---

## ✅ Case-M1 · iOS 删除二次确认（10+ 年仍站得住）

**场景**：iOS 邮件 App 删除单封邮件

**决策**：滑动即删，不弹"确定要删除吗？"对话框。改为顶部出现「撤销」横条（5 秒）。

**哲学依据**：
- `[康德 · 绝对命令]`：每个用户每次删除前都被打断 → 不能成为普世法则
- `[Canon-D5]`：好设计是隐而不见的 → 打断 = 显眼 = 失败
- `[Canon-D10]`：尽可能少的设计 → 用「撤销」代替「确认」

**为什么经典**：把"防错误"从**前置阻塞**改为**后置可恢复**，是从工业时代到数字时代的根本范式转换。

---

## ✅ Case-M2 · Gmail Compose 抽屉式（而非全屏弹窗）

**场景**：Gmail 写新邮件

**决策**：右下角浮起一个小抽屉，可最小化、可后台保留，不阻塞主界面浏览。

**哲学依据**：
- `[庄子 · 庖丁解牛]`：顺着用户的工作流，不切断主线
- `[Canon-D5]`：写邮件时仍能看收件箱 → UI 隐而不见
- `[Canon-D2]`：可用性 = 不强制专注

---

## ✅ Case-M3 · Linear 命令面板（Cmd+K）

**场景**：Linear 中任何操作（创建 issue / 切换视图 / 跳转项目）

**决策**：用一个统一的命令面板，键盘驱动，无菜单层级。

**哲学依据**：
- `[奥卡姆剃刀]`：所有动作 → 一个入口 → 删除冗余菜单
- `[Canon-D10]`：少即是多
- `[Canon-B3]`：Less is more

---

## ❌ Case-M4 · 全屏强制广告弹窗

**场景**：用户打开 App，弹出全屏运营海报，3 秒后才能关闭

**违反**：
- `[Canon-D5]`：极度突兀
- `[Canon-D6]`：诚实——你打开 App 不是为了看广告
- `[Canon-D10]`：典型 over-design

**修正方向**：
- 改为顶部不阻塞 banner（可关闭）
- 或集成进首页 feed（用户可滑过）
- 必须做的运营信息 → 信息中心红点 + 用户主动点击

---

## ❌ Case-M5 · 弹窗叠弹窗

**场景**：用户点击删除 → 弹"是否删除" → 点确认 → 又弹"此操作不可恢复" → 点继续 → 再弹"最后确认"

**违反**：
- 极致违反奥卡姆剃刀
- 用户被惩罚式确认，体验到的是"你不被信任"
- `[Canon-D6]`：不诚实——把开发者的恐惧外化为用户的负担

**修正方向**：
- 单次确认 + 可撤销机制
- 如真的不可恢复，用一次"输入名称确认"，但只一次

---

## ❌ Case-M6 · 弹窗里嵌入完整表单（10+ 字段）

**场景**：点"新增订单"弹出弹窗，弹窗内有 12 个字段的长表单

**违反**：
- `[Canon-D5]`：弹窗本应承载快速操作，长表单 = 误用
- `[庖丁解牛]`：硬切割了用户的填写心流
- 移动端会被键盘遮挡

**修正方向**：
- 改为页面级（路由跳转）或抽屉式（侧滑）
- 弹窗只承载 ≤ 3 个字段的快速操作

---

> **判定原则**：你想加弹窗时，先回答这 3 个问题：
> 1. 不能用 toast / inline / sidebar 代替吗？
> 2. 这个动作真的需要打断用户吗？
> 3. 这个动作不可逆吗？如果可逆 → 改用「撤销」模式
